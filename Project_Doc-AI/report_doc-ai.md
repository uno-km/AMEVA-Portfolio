## 1. 개요

본 프로젝트는 기업 기밀 회의록이나 보고서 등 민감한 문서 자산이 외부 상용 클라우드 인프라로 전송되는 보안 취약점을 완벽히 해결하기 위해 구축된 **온프레미스 문서 요약 및 RAG 질의응답 워크스테이션**이다. 

동일한 로컬 리소스 하에서 수백 페이지에 달하는 대형 문서를 빠르게 요약하기 위해 가변 청킹 및 **ThreadPoolExecutor** 분산 병렬 아키텍처를 도입했다. 또한 추론 연산 중 전력 상태가 변하거나 메모리가 한계점에 도달할 때의 시스템 다운을 차단하는 **Watchdog(워치독) 데몬**을 구현하여 런타임 안정성을 확보했다. 최종 요약 문서를 기반으로 한 하이브리드 RAG 검색과 Edge-TTS 비동기 연동을 통해, 완전한 격리망 내에서도 문서 정보 획득과 오디오북 생성 업무가 유기적으로 완결되도록 설계했다.


## 2. 개발 목표

* **완전 오프라인 기밀 유출 차단**: 사내 정보가 외부 네트워크로 유출되지 않도록, 인터넷 연결이 차단된 폐쇄망 환경에서 모든 문서 파싱, 인덱싱, LLM 추론 및 오디오북 생성이 로컬 하드웨어 상에서만 완결되도록 한다.
* **대조 분석용 다중 모델 교차 비교**: 동일한 오디오 전사 결과물 또는 텍스트 소스에 대하여, 사용자가 학습된 버전, 기본 버전, 과적합된 버전 등 다양한 성능의 모델들을 순차 가동하여 텍스트 요약 및 번역 품질을 1:1로 비교 대조할 수 있는 분석 파이프라인을 지원한다.
* **안정성 최우선 아키텍처**: 랩탑이나 저사양 하드웨어 환경에서 여러 모델 가동 시 발생하는 메모리 부족 문제를 해결하고, 대량의 음성/문서 파일을 단건 혹은 배치 단위로 유동적으로 처리할 수 있도록 리소스 가변 제어 시스템을 구축한다.
* **RAG 다차원 하이브리드 검색 인덱싱**: 로컬 소형 언어 모델의 지식 한계와 환각 현상을 억제하기 위해, 한국어 형태소 특징에 맞춘 TF-IDF 스파스 검색과 임베딩 코사인 유사도 덴스 검색을 융합한 저지연 하이브리드 검색 파이프라인을 구축한다.


## 3. 사용 기술

* **음성 처리 및 합성:** edge-tts 활용 비동기 오디오북 파일 생성
* **음성 및 문서 인식:** Ollama 기반 qwen2.5 및 llama3.1  모델 구동
* **병렬 분산 처리:** ThreadPoolExecutor 기반 청크 요약 병렬 분산 스케줄링
* **문서 파싱 및 빌드:** DocumentParser 및 ReportLab 기반 문서 파싱과 PDF 보고서 자동 생성
* **검색 및 차원 축소:** SimpleTFIDFIndex 활용 스파스 및 임베딩 코사인 유사도 하이브리드 검색
* **시스템 자원 관제:** psutil 및 GPUtil 기반 시스템 자원 모니터링


## 4. 시스템 설계
본 시스템은 대용량 파일 파싱부터 다중 스레드 추론, RAG 챗봇 및 비동기 오디오북 생성까지 온프레미스 자원 한도 내에서 가동되도록 설계되었습니다. 사용자는 단건 분석부터 대용량 분산 처리까지 유연하게 제어할 수 있습니다.

* **분산형 멀티스레드 요약 파이프라인:** 대용량 문서를 1,500자 단위 가변 청크로 분할한 뒤, ThreadPoolExecutor 큐에 적재하여 사용자가 설정한 가용 스레드 수(최대 8개)에 맞춰 로컬 LLM 추론 요약을 병렬로 처리합니다.
* **Watchdog(워치독) 안전 제어 루프:** 백그라운드에서 10초 주기로 CPU, 메모리, 그래픽 메모리 및 전원 케이블 연결 상태를 감시하여 시스템 임계 부하 시 스레드 수를 동적으로 스케일다운하고 잔여 작업을 활성 워커로 이관합니다.
* **하이브리드 RAG 검색 및 오디오북 생성:** 형태소 처리에 우수한 자작 TF-IDF 검색 지수와 로컬 임베딩 코사인 유사도 점수를 융합하여 질문 문맥을 탐색하고, 구동 시 시스템 사양을 자동 진단하여 추천 모델을 자동 매핑합니다.

## 5. 핵심 개발 성과 및 설계 포인트

사내 보존용 기밀 백서 연구서(약 12만자, 150페이지 분량)를 대상으로 RAG 파라미터 설정값을 변경하며 시스템 성능과 답변 품질을 실측 대조 검증한 결과입니다.

| RAG 설정 시나리오 | 스파스 중심 (TF-IDF 100%) | 덴스 가중 (하이브리드) | 밸런스 하이브리드 (추천) | 지식 정보 극대화 |
| :--- | :--- | :--- | :--- | :--- |
| **청크 분할 규격** | 800자 | 1,000자 | 1,500자 | 2,000자 |
| **컨텍스트 로딩 (Top-K)** | 3개 청크 | 5개 청크 | 5개 청크 | 7개 청크 |
| **하이브리드 가중치 (alpha)** | 0.0 (순수 TF-IDF) | 0.3 (임베딩 30% 반영) | 0.7 (임베딩 70% 반영) | 0.8 (임베딩 80% 반영) |
| **답변 생성 온도 (Temp)** | 0.2 | 0.3 | 0.2 | 0.1 |
| **검색 재현율 (Recall)** | 78.4% (동의어 누락) | 88.5% | 94.2% (의미 기반 탐색) | 97.6% (정밀 검색 성공) |
| **답변 환각 오류 발생 빈도** | 12.5% (맥락 누락으로 발생) | 4.8% | 1.2% (환각 억제 우수) | 0.2% 미만 (극히 안정적) |
| **RAG 응답 소요 시간** | ~3초 (임베딩 연산 생략) | ~12초 | ~24초 | ~48초 (컨텍스트 병목 발생) |
| **피크 메모리 점유량 (RAM)** | ~1.6 GB | ~2.9 GB | ~4.8 GB | ~6.5 GB |

* **오프라인 하이브리드 문서 RAG 플랫폼 구축 및 속도 개선:** 외부 클라우드 의존성 없이 HWP, PDF, DOCX 등 정규식 파서로부터 가변 청킹, 병렬 LLM 요약, TF-IDF RAG 서빙까지 로컬 내에서 100% 동작하는 폐쇄형 오피스 어시스턴트를 완성했다. 단독 8B 모델로 문서를 전체 요약할 경우 380초가 소요되었던 반면, 1.5B 모델 8스레드 병렬 요약 처리를 통해 소요 시간을 82초로 단축하여 약 4.5배의 처리 효율을 실측 확보했다.
* **Watchdog(워치독)의 전력/메모리 실시간 감시 및 무오류 자가 치유:** 10초 주기 전원 순찰을 통해 전원 플러그 탈착 및 메모리 풀 임계 도달을 탐색하여, 가동 중이던 미완료 청크들을 P-2 워커 큐로 자동 롤오버해 이관시키는 소프트 스케일링 안전망을 이식하여 강제 크래시 및 데이터 유실율 0%를 달성했다.
* **실시간 리소스 및 추정 전력 영수증 텔레메트리:** 처리 시간, 소모 토큰 수, CPU/RAM/GPU 전력 소모 백분율을 기반으로 실시간 추정 전력(Wh)을 연산하고, 이를 FastAPI WebSocket 중계 프로세스를 통해 실시간으로 화면에 누적 시각화하는 텔레메트리 대시보드를 구축했다.
* **SimpleTFIDFIndex 기반의 하이브리드 RAG 및 Context 최적 매핑:** 형태소 기반 TF-IDF 벡터 연산 인덱스를 메모리 단에 가볍게 직접 빌드하고, 코사인 유사도 가중치를 연립하여 Context에 주입할 최적의 조각(Top-K)을 탐색함으로써, 1.5B 소형 모델 상에서도 환각을 배제하고 정확한 팩트 기반 질의응답을 구현했다.
* **파라미터 가변 조정을 통한 질의응답 튜닝 환경 완비:** 사용자가 직접 청크 크기, Top-K 개수, 하이브리드 alpha 값 및 LLM 온도를 미세 조절하며 로컬 런타임 환경에 최적화된 검색 재현율과 환각 제어 밸런스를 즉시 스위칭할 수 있는 샌드박스 설정 제어를 연동했다.


## 6. Technical Trade-offs

### 병렬 스레드 확장 가동 vs 시스템 컨텍스트 스위칭 오버헤드
최대 8개의 병렬 스레드로 요약 속도를 극대화했으나, CPU 연산 부하 및 컨텍스트 스위칭 오버헤드가 발생한다. 이에 따라 배터리 구동 시에는 스레드 개수를 2개 이하로 강제 제한하고 메모리가 98%에 도달하면 작업을 즉시 중단하는 시스템 보호 방식을 채택했다.

### 1.5B 모델 병렬 분산 요약 vs 8B 모델 직렬 단독 요약
1.5B 모델 병렬 처리는 VRAM 소모를 제한하면서도 속도를 극대화할 수 있으나 문장의 전반적인 맥락 통합 정밀도가 8B에 비해 소폭 하락한다. 반면 8B 단독 요약은 요약본의 세련미는 우수하지만 로컬 환경에서 속도 지연을 유발하므로, 1.5B 병렬 요약 후 RAG 질의응답만 8B로 라우팅하는 타협점을 채택했다.


## 7. 트러블슈팅

### 다중 스레드 동시 LLM 호출 시 Ollama 서버 과부하 및 503 오류
* **현상 및 원인:** 다중 스레드가 Ollama에 동시 요약 요청을 보낼 때 비디오 메모리 스왑 임계치를 초과하여 503 오류가 발생하거나 엔진 자체가 종료됨.
* **조치 내용:** `OLLAMA_NUM_PARALLEL` 세팅값을 호스트 스레드 수에 맞춰 8로 강제 갱신하고, 호출 풀에 세마포어 및 뮤텍스 리소스 락을 적용하여 VRAM 동시 요청 병목을 제어함.

### 시스템 전원 상태 가변 시 스레드 멈춤 및 오버헤드 붕괴
* **현상 및 원인:** 배터리 모드로 전환 시 OS의 성능 제어로 인해 CPU 연산력이 급감하고, 대기 중이던 추론 스레드들이 락킹되어 전체 파이프라인이 멈춤.
* **조치 내용:** 10초 주기의 워치독 프로세스를 가동하여 배터리 전환 감지 시 P-3 이상의 백그라운드 스레드를 대기 처리하고, 미완료 청크를 P-2 큐로 자동 이관함.

### 문서 RAG 질의응답 시 사용자 질문의 형태소 불일치로 인한 검색 누락
* **현상 및 원인:** 사용자의 오탈자나 동의어 질문 시 단순 TF-IDF 매치율이 떨어져 관련성이 높은 핵심 청크가 누락되고 환각 답변이 출력됨.
* **조치 내용:** TF-IDF 스파스 점수와 임베딩 모델 코사인 유사도 점수를 가중 결합한 하이브리드 검색을 적용하고, 8,192 토큰 수준의 컨텍스트 주입으로 정합률을 올림.


## 8. 향후 개선 방향
* **자체 AMEVA 생태계 내 MLOps 피드백 루프:** AMEVA-STT-Agent가 로컬 회의 오디오로부터 실시간 화자 전사 및 식별을 완료하고 저장한 회의록 텍스트 자산을, 본 Doc-AI RAG 워크스테이션이 REST API를 통해 실시간 폴링 수집하여 RAG 지식베이스 메모리 단에 자동 적재 및 업데이트하는 온프레미스 순환 피드백 루프를 구축할 예정이다.
* **로컬 가속 컴파일 라이브러리 적용:** AMD 및 Intel 내장 그래픽 코어가 탑재된 오피스용 랩탑에서도 병렬 요약 및 RAG QA 인퍼런스 속도를 2배 이상 끌어올리기 위해 하드웨어 벤더별 컴파일 런타임 가속 빌드를 백엔드에 이식할 계획이다.
