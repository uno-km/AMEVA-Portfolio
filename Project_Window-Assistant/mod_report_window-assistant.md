# AMEVA Window Assistant
*로컬 PC 환경에서의 음성/시각/언어 모델별 실측 추론 품질 및 비교 분석*

## 1. 개요 및 목적
본 포트폴리오는 사내 기밀 및 개인 화면 정보가 외부 클라우드 서버로 상시 유출되는 보안 취약점 해결을 목적으로 개발된 AMEVA Window Assistant의 핵심 개발 성과를 요약합니다. 네트워크 연결이 완전히 차단된 폐쇄망 환경에서도 로컬 기기 단독으로 동작하는 독립 실행형 인공지능 비서를 성공적으로 구현했습니다.

## 2. 사용 모델 및 사용 언어
- **음성 인식 (STT)**: Whisper.cpp (경량화 양자화 모델)
- **의도 분류 라우터**: Qwen2.5-0.5B-Instruct-Q4
- **시각 상황 인지 (VLM)**: Qwen2-VL-2B-Instruct-Q4, Qwen2-VL-7B-Instruct-Q4
- **로컬 메인 언어 모델 (LLM)**: Llama-3.1-8B-Instruct-Q4_K_M (Ollama 런타임)
- **사용 기술**: Python, Tkinter, Docker Compose, Windows SAPI

## 3. 핵심 트러블슈팅 및 해결

1) 시각 인지 모델 지연 및 메모리 부족 해결
- **현상 및 원인**: 고해상도 이미지 상시 분석 시 10초 이상 지연되고 그래픽 메모리가 초과되어 프로그램이 강제 종료됨.
- **조치 내용**: 초경량 의도 분류 모델을 전면에 배치하여 화면 분석이 필요 없는 질문은 이미지 처리를 생략함.

2) AI 연산 시 GUI 스레드 블로킹 (화면 멈춤)
- **현상**: 단일 프로세스 내에서 대용량 AI 모델의 추론 연산이 실행되는 동안 Tkinter GUI 메인 루프가 멈춰 사용자 창이 백화되거나 조작 불가능 상태가 됨.
- **해결책 (비동기 리소스 격리 오케스트레이션)**: LLM/VLM/Router 추론 서버를 Docker Compose 기반의 백그라운드 독립 컨테이너 레이어로 완전히 분리 격리. GUI 프로세스 측은 Producer-Consumer Queue 모델을 채택한 데몬 스레드 워커를 구현해 비동기 통신을 처리하도록 설계.
- **결과**: 백그라운드 추론 연산 중에도 마이크 볼륨 실시간 시각화 및 UI 렌더링을 60fps 프레임 드랍 없이 유지. 또한 처리하는 중에도 새로운 사용자의 요청 큐를 추가로 등록하여 끊김 없이 후속 작업을 대기 및 처리할 수 있도록 시스템을 개선함.

3) 대기 상태의 불필요한 자원 낭비 해결
- **현상 및 원인**: 마이크 대기 시 무음 구간에서도 음성 인식기가 작동하여 CPU 점유율이 치솟음.
- **조치 내용**: 입력 소음 수준을 계산하여 발화 순간에만 인식기를 작동시키도록 제어함.
