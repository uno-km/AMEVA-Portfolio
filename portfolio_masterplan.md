## Ⅱ. 프로젝트별 상세 명세 (Draft)

### 1. AMEVA-Window-Assistant (로컬 멀티모달 데스크탑 어시스턴트)
* **목적 및 문제 정의**:
  * Copilot이나 Gemini Screen Sharing 비서와 같이 화면을 공유하며 협업하는 AI를 완벽한 오프라인 로컬망(Intranet)에 이식하고자 함.
  * 초거대 VLM(시각 언어 모델)을 로컬 GPU에서 상시 가동하면 VRAM 부족 및 FPS 저하로 메인 루프가 블로킹되는 병목이 발생.
* **기술적 의사결정 (Trade-off)**:
  * **VLM 대신 OCR-LLM 라우팅**: 모든 화면을 VLM으로 분석하지 않고, Tesseract 기반 Bounding Box 밀도가 충분할 경우 OCR 데이터로 추출하여 경량 LLM(Llama-3.1-8B)에 전달하고, 복잡한 이미지나 시각 전역 맥락만 VLM(Qwen2-VL-2B)으로 선택적 라우팅하여 추론 병목을 약 70% 감소시킴.
  * **OS Direct PCM Capture**: FFmpeg를 사용하여 오디오 포맷을 사후 리샘플링하는 오버헤드를 줄이기 위해, `sounddevice`로 OS 오디오 장치에 직접 16kHz PCM 형태로 버퍼 인터럽트 수집하여 오디오 I/O 연산 오버헤드 0으로 소거.
* **트러블 슈팅**:
  * **GUI 루프 프리징**: LLM 추론 시 Tkinter 화면이 응답 없음으로 굳어버리는 에러. Producer-Consumer 비동기 스레드 큐를 구축하여 메인 루프를 백그라운드 워커와 스레드 안전하게 격리하여 해결.
  * **배경 소음으로 인한 STT 무한 대기**: `RMS` 에너지를 100ms 단위로 모니터링하여 임계값 이하 무음 발생 시 타이머 만료 강제 컷오프 로직 구현.
* **성과 및 셀프 피드백**:
  * 완전 오프라인 환경 구동 성공. 다만 STT-LLM-VLM-TTS 파이프라인 구동 시 RAM/VRAM 점유율이 12~14GB 이상을 기록하며 End-to-End 지연 시간이 5~8초 소요되어 저스펙 PC에서의 하드웨어 연산 한계점을 수치적으로 규명함.

### 2. AMEVA-Agent-Orchestra (자율형 멀티 에이전트 시스템)
* **목적 및 문제 정의**:
  * Claude Code처럼 CLI나 백그라운드에서 파일과 명령어를 제어하며 스스로 코딩하고 디버깅하는 시스템 구현.
  * 로컬 LLM의 컨텍스트 한계 및 에이전트 간 역할 충돌(PM, 개발자, QA)로 인한 컨텍스트 포화 및 데드락 발생.
* **기술적 의사결정 (Trade-off)**:
  * **Guided Generation**: 자연어 출력 대신 LlamaGrammar로 JSON 스키마를 강제 생성하게 함으로써 파싱 오류 차단.
  * **자원 격리 및 관제**: 에이전트마다 스레드를 분리하되 `psutil`/`GPUtil` SRE 모니터링 데몬을 띄워 락킹 충돌 및 VRAM 누수를 2초 주기로 관제.
* **트러블 슈팅**:
  * **다중 스레드 llama.cpp 세그먼테이션 폴트**: 여러 에이전트가 단일 로컬 LLM 서버에 동시 추론 요청 시 Crash 발생. `threading.Lock`을 LLM 호출 인터페이스에 적용해 Mutex 제어로 해결.
  * **JSON 구문 에러로 인한 루프 크래시**: LLM이 JSON 중괄호를 덜 닫는 등 구문 에러. 중괄호 스택 매칭 기반 `Hybrid Parser`를 제작하여 예외 발생 시 문자열을 2차 복구하는 방어 로직 설계.
* **성과 및 셀프 피드백**:
  * 자율 디버깅 루프의 흐름 제어 성공. 그러나 격리 샌드박스의 파일 권한 예외 및 무한 루프 진입 시 180초 지연 데드락 회수 처리 로직의 복잡성 증대로 인한 로컬 엣지 연산 비용 확인.

### 3. AMEVA-Dead-Internet-Threatre (잠재적 인격 동역학 시뮬레이션 시스템)
* **목적 및 문제 정의**:
  * AI 에이전트들이 폐쇄망 커뮤니티 내부에서 자유 논쟁 및 여론을 생성할 때, 극단 수렴 현상 및 대화 정체를 방지하기 위한 통제 기법 연구.
* **기술적 의사결정 (Trade-off)**:
  * **Director Agent 배치**: 에이전트들이 독자적으로 소통하도록 두는 대신, 감독(Director) LLM 에이전트가 중간에 개입하여 여론 델타(Delta) 섭동을 조절하는 중앙 통제 아키텍처 채택.
* **트러블 슈팅**:
  * **3B 이하 SLM 모델의 기억 포화**: 대화가 지속될수록 SLM의 컨텍스트 윈도우 한계로 정체성 붕괴 발생. 대화 이력을 감정 수치 벡터 `LPDE` 인격 상태 벡터로 수평 요약 압축하여 메모리 크기를 최소화하여 완화.
* **성과 및 셀프 피드백**:
  * 완전 자율 소통보다 감독 에이전트가 개입하는 구조가 여론 교착을 푸는 핵심 열쇠임을 통계적으로 실증함.

### 4. AMEVA-Benchmark-Suite (LLM 벤치마킹 플랫폼)
* **목적 및 문제 정의**:

* **기술적 의사결정 (Trade-off)**:
  * **Smart SWAP 격리**: 테스트 후 VRAM 찌꺼기를 확실히 날리기 위해 Ollama 서비스를 단순히 끄는 게 아니라 컨테이너 전체를 완전히 파괴하고 새로 생성하여 100% 결정적(Deterministic) 클린 스테이트 유지.
* **트러블 슈팅**:
  * **판정관 모델과 테스트 모델 VRAM 충돌**: 8B급 모델 두 개를 한 번에 올리면 OOM 크래시 발생. 테스트 모델 컨테이너 가중치를 메모리에서 완전히 Unload한 후 판정관(EXAONE-3.5)을 순차 구동하는 격리 런타임 스케줄링으로 해결.
* **성과 및 셀프 피드백**:

### 7. AMEVA-STT-Agent (Whisper-Vosk 하이브리드 화자 분리 및 전사 엔진)
* **목적 및 문제 정의**:
  * 로컬 환경에서 긴 음성 파일이나 회의록에 대해 화자별로 대화를 분리(Diarization)하고 텍스트화하고자 함.
* **기술적 의사결정 (Trade-off)**:
  * **Whisper와 Vosk의 하이브리드**: Whisper는 텍스트 디코딩 정확도가 높으나 화자 식별 기능이 약하고, Vosk는 X-Vector 임베딩을 통해 화자 식별 및 세그먼팅 속도가 빠름. 따라서 두 모델의 시간 차 인터벌을 시간차 최소화 정렬 알고리즘으로 동적 매핑하여 장점만 결합.
* **트러블 슈팅**:
  * **대용량 파일 오디오 벡터 연산 시 CPU 과부하**: 화자 간 임베딩 구분을 위한 K-Means 군집화 시 멱반복법(`Power Iteration`) 기반 `PCA` 축소를 가동하여 대량 텐서의 CPU 연산 병목 완화.
* **성과 및 셀프 피드백**:
  * 외부 API(Clara, VITO 등) 없이 로컬 단독 화자 분리 오프라인 서비스 구축.

### 8. AMEVA-STT-Trainer (Windows CPU 최적화 Whisper LoRA 파인튜닝 플랫폼)
* **목적 및 문제 정의**:
  * 로컬 윈도우 환경에서 특정 도메인(통화 녹음, 특정 방언 등)에 특화되도록 Whisper 음성 인식 모델을 LoRA 파인튜닝함.
* **기술적 의사결정 (Trade-off)**:
  * **IterableDataset 스트리밍**: 윈도우 디스크 I/O 병목 및 메모리 캐싱 오버헤드를 막기 위해 학습 데이터를 디스크에 다 올리지 않고 필요할 때 실시간 스트리밍 디코딩 로드 처리.
* **트러블 슈팅**:
  * **WinError 87 (매개변수가 틀립니다)**: Windows 환경에서 멀티프로세싱 데이터 로딩 시 PyTorch dataloader와 ctypes 간 충돌 발생. `pin_memory=False` 처리 및 `num_workers=0`으로 세팅하여 윈도우 커널 충돌 우회 해결.
* **성과 및 셀프 피드백**:
  * 저사양 Windows 환경에서 메모리 고장 없이 Whisper 파인튜닝 파이프라인 완수.
